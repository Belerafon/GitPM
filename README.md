# GitPM

Git-first система управления проектами и задачами с веб-интерфейсом, GitLab Merge Request workflow и поддержкой агентов.

## Текущий статус

Репозиторий находится на стадии планирования. Программная реализация еще не начата.

Документы:

- `docs/GitPM_Implementation_Plan_v0.2.md` - архитектура, формат данных, API и продуктовые решения;
- `docs/GitPM_Work_Plan_v0.1.md` - этапы разработки, quality gates, верификация и обязательные E2E-сценарии;
- `docs/PROGRESS.md` - живой журнал статуса, commit, проверок, блокировок и следующего шага.

## Как вести работу

1. Выбрать текущий этап в `docs/PROGRESS.md`.
2. Выполнять задачи и проверки из соответствующего раздела `GitPM_Work_Plan_v0.1.md`.
3. Записать commit SHA, команды тестов и результат ручной приемки в файл прогресса.
4. Переводить этап в `done` только после выполнения всех критериев выхода.
5. Изменения архитектуры сначала фиксировать в версии Implementation Plan или отдельном ADR.

## Основные принципы

- Git является источником истины.
- Каждая сущность хранится в отдельном YAML-файле.
- Пользователи и агенты работают в отдельных ветках и Git worktree.
- Изменения проходят через commit, push и GitLab Merge Request.
- Архивирование и физическое удаление являются отдельными операциями.
- Восстановление и откат выполняются штатными средствами Git.
- Собственная база данных и отдельный механизм Undo в первой версии не используются.
- Прогресс подтверждается тестами и доказательствами, а не только процентом готовности.
